## 1. 产品定位与目标

### 1.1 一句话描述

Bonne-Santé 是女性专属的原生 iOS 健康助手：在 CalorieCop 饮食记录能力之上，融合体检报告、Apple Watch 数据与月经周期，形成统一的个人健康画像，并据此生成减脂计划、风险预警与就诊待办。

> **平台说明**：本产品为**原生 iOS App**（非微信小程序）。HealthKit、Vision、EventKit 等系统能力均依赖原生架构，基于 `kaka-jun/CalorieCop` 改造是正确路线。

### 1.2 第一优先级（本 PRD 最高原则）

**健康数据整合必须与 CalorieCop 减脂能力深度融合，而非两套功能简单并列。**

具体含义：
- 体检异常指标 → 影响每日热量预算与蛋白质下限（如肾功能异常时调整蛋白建议）
- Apple Watch 运动数据 → 影响 TDEE 与周训练完成度
- 减脂目标（体重/体脂/期限）→ 与饮食日记、热量环、AI 顾问共享同一数据源
- 高危风险与复查计划 → 自动流入待办，并在首页仪表盘展示
- 月经周期 → 调节训练强度建议与饮食 tips（阶段三深度联动）

所有页面读取的数据，必须来自统一的 **`UnifiedHealthContext`**，禁止各 Tab 各自维护孤立状态。

### 1.3 核心价值

- 把分散在相册、HealthKit、饮食日记中的数据，收敛为一个**可行动的健康仪表盘**。
- 保留 CalorieCop「拍照识食 + 自然语言录入 + AI 顾问」的日常使用习惯，并用个人体检数据让它更懂你。
- 根据女性生理周期给出贴身运动与饮食建议，而非通用健身 App 模板。
- 以莫兰迪色系 + 系统原生组件，达到 Apple 级视觉与交互质感。

### 1.4 目标用户

25–45 岁、有定期体检需求、佩戴 Apple Watch 的女性；无需医学背景，但需要可信赖的风险提示与复查提醒。

### 1.5 技术基座

| 项目 | 说明 |
|------|------|
| 基座仓库 | `https://github.com/kaka-jun/CalorieCop` |
| 技术栈 | SwiftUI、SwiftData、HealthKit、Vision、EventKit、UserNotifications |
| 最低版本 | iOS 17.0+、Xcode 15.0+ |
| AI 能力 | **DeepSeek**（文本）+ **Qwen VL**（图像），用于食物录入、顾问与可选报告结构化辅助 |
| 开发环境 | Windows Cursor → SSH 远程 Mac mini 编译，远程桌面预览 |

## 2. CalorieCop 保留清单（改造时禁止删除）

以下功能为**必须保留并增强**，改造时只允许重构 UI 与数据接入，不允许砍掉核心能力。

| 功能 | 原模块路径 | Bonne-Santé 改造要求 |
|------|-----------|---------------------|
| 自然语言食物录入（中文） | `Services/AIService/DeepSeekService.swift` | DeepSeek API；保留 `FoodParsingPrompt.swift` |
| 拍照识食 | `Views/FoodInput/` + Qwen VL | **保留不变**；确认页展示当日剩余热量预算 |
| 食物偏好快捷录入 | `Models/FoodPreference.swift` | 保留；减脂 Tab 内可达 |
| 饮食日记与营养卡片 | `Views/Dashboard/`、`Components/NutritionCard.swift` | 保留；合并进「减脂」Tab |
| 热量收支展示 | `Components/CalorieBalanceView.swift` | 升级为首页环形图 + 收支明细 |
| 目标体重设定 | `Models/UserGoal.swift` | **扩展**：增加目标体脂率、期望达成日期、当前体脂（手动/HealthKit） |
| AI 顾问对话 | `Views/History/AIAdvisorView.swift` | 改造为 **DeepSeek** 流式对话；Prompt 注入体检摘要 + 减脂进度 + 周期阶段 |
| 历史记录 | `Views/History/HistoryView.swift` | 保留；与饮食日记合并或作为减脂子页 |
| API Key 与应用内配置 | `Services/APIKeyManager.swift`、`Views/Settings/APIKeySetupView.swift` | 升级为 `AISettingsView`；**DeepSeek Key + Qwen Key**；Keychain 存储；连接测试 |
| HealthKit 基础同步 | `Services/HealthKitService.swift` | **扩展**：增加锻炼类型、经期数据、体重体脂读取 |
| SwiftData 本地存储 | `Models/FoodEntry.swift` 等 | 保留并扩展新 Model，不替换持久化方案 |

### 2.1 融合改造规范

```
改造前（CalorieCop）          改造后（Bonne-Santé）
─────────────────────        ─────────────────────────────
UserGoal（仅体重）      →    UserGoal + BodyComposition + TargetTimeline
HealthKit（消耗）       →    HealthKit（消耗 + 锻炼 + 经期 + 体重）
FoodEntry（孤立）       →    FoodEntry ← UnifiedHealthContext.dailyBudget
DeepSeekService（文字）  →    唯一文字 AI，OpenAI 兼容格式
Qwen VL（图像）         →    保留不变
AI Advisor（仅饮食）    →    AI Advisor + healthSummary + riskFlags 上下文
Dashboard（热量）       →    Dashboard（热量 + 风险 + 待办 + 周期条）
```

**新增核心服务（不可省略）：**

- `UnifiedHealthContext`：聚合所有健康数据的单一真相源（Single Source of Truth）
- `HealthProfileEngine`：读取体检指标，输出约束与建议（如蛋白上限、复查提醒）
- `IntegratedTDEEEngine`：TDEE = HealthKit 消耗 + 体检相关调整 + 减脂目标赤字
- `WorkoutPlanEngine`：按周期阶段生成结构化周训练计划

## 3. 全局设计规范

### 3.1 设计哲学

- **做减法**：每屏只突出一个核心动作或信息层级。
- **苹果原生触感**：系统字体、SF Symbols、Material 材质优先；自定义仅用于莫兰迪色板。
- **色彩疗愈**：低饱和莫兰迪色，适合长期每日打开。
- **数据可信**：健康数据展示带来源标签（体检 / Watch / 手动 / AI 估算）。

### 3.2 色板（莫兰迪主题）

| 角色 | 色值 | 用途 |
|------|------|------|
| 主色（温柔蓝） | `#ADC8F5` | 主按钮、选中态、环形图进度 |
| 背景色 | `#F9FBFF` | 页面背景 |
| 强调色（粉紫） | `#E5A5CF` | 周期相关、重要提醒、待办标记 |
| 警示色 | `#E8A0A0` | 高危风险（低饱和红，非系统纯红） |
| 文字主色 | `#1C1C1E` | 标题、正文 |
| 文字辅色 | `#6E6E73` | 说明、单位 |
| 卡片底 | `.regularMaterial` | 磨砂卡片，圆角 16pt，无硬阴影 |

**周期主题色（阶段三启用）：** 经期暖粉 / 卵泡期淡绿 / 黄体期薰衣草紫，通过 `@Environment(\.cyclePhase)` 注入。

### 3.3 字体、图标与布局

- 字体：San Francisco + Dynamic Type（`.body`、`.title2`、`.largeTitle`）
- 图标：SF Symbols 5 only
- 圆角：卡片 16pt、按钮 12pt、输入框 10pt
- 间距：4 的倍数网格，水平边距 20pt

### 3.4 Apple HIG 合规（全阶段持续）

- 支持浅色 / 深色模式（莫兰迪色板需提供 Dark 变体）
- Dynamic Type 不截断关键数字
- VoiceOver 为图表、环形图提供 accessibilityLabel
- 尊重 Safe Area；Tab Bar 使用系统原生样式
- 敏感操作（删除报告、导出数据）需二次确认

### 3.5 导航架构（5 Tab，阶段一起骨架到位）

| Tab | 标题 | SF Symbol | 核心内容 |

## 3.6 AI 模型接入与 API Key 配置（新增专章）

> **文字 AI 仅使用 DeepSeek**，不使用 MiniMax。Qwen VL 继续负责拍照识食。

### 3.6.1 设计原则

| 原则 | 说明 |
|------|------|
| **BYOK** | 用户自带 API Key，无 Bonne-Santé 后端，隐私与成本可控 |
| **渐进配置** | 仅配 **DeepSeek** 即可启用核心饮食 + AI 顾问；Qwen 为拍照识食增强项 |
| **能力可见** | 用户始终清楚「哪个 Key 驱动哪个功能、当前是否可用」 |
| **优雅降级** | 未配置或离线时，不阻断 App，引导至手动路径 |
| **安全存储** | Key 存 Keychain（升级自 CalorieCop 的 UserDefaults），仅本机解锁可读取 |

### 3.6.2 AI 能力矩阵

| 功能 | 所用模型/技术 | 所需 Key | 联网 | 未配置时的降级 |
|------|--------------|----------|------|----------------|
| 自然语言食物录入 | DeepSeek 文本 | **DeepSeek（必须）** | 是 | 跳转手动营养录入表单 |
| AI 健康顾问 | DeepSeek 文本 | **DeepSeek（必须）** | 是 | 只读历史对话，禁用发送 |
| 拍照识食 | Qwen VL 图像 | Qwen（推荐） | 是 | 提示切换文字录入或手动填写 |
| 体检报告 OCR | Apple Vision（设备端） | 无 | 否 | 手动逐项输入指标 |
| 报告结构化辅助（阶段二，可选） | DeepSeek 文本 | DeepSeek | 是 | 仅用 Vision 结果 + 用户校对 |
| 门诊截图解析（阶段三，可选） | Vision + DeepSeek | DeepSeek | 混合 | 手动创建待办 |

**关键结论：**
- **DeepSeek Key = 核心 AI 能力钥匙**（饮食解析 + 顾问 + 可选报告辅助）
- **Qwen Key = 视觉增强钥匙**（拍照识食，**保留 CalorieCop 原方案不变**）
- **体检 OCR 主体不依赖 AI Key**，降低阶段二门槛；LLM 辅助为可选项，默认关闭，用户在设置中主动开启

### 3.6.3 配置入口（多触达，避免藏在深处）

```
触达优先级：
1. 首次引导 Onboarding Step 3「连接 AI 服务」（可跳过，标注功能受限）
2. 功能拦截卡片 APIKeyPromptView（饮食录入 / AI 顾问 / 可选报告辅助）
3. 我的 → AI 服务（主配置页，随时修改）
4. 首页状态条（仅当 DeepSeek 未配置时显示橙色提示条，可关闭）
```

### 3.6.4 AI 服务设置页 UI 规格（升级 `APIKeySetupView` → `AISettingsView`）

**页面类型**：工具型设置页（Tool Page）—— 任务明确、低干扰、步骤清晰。

**布局结构（上 → 下）：**

1. **服务状态总览卡**（新增）
   - 三行能力状态灯：`饮食解析` / `AI 顾问` / `拍照识食`
   - 每行：SF Symbol 图标 + 功能名 + 状态徽章（`已就绪` 绿 / `未配置` 橙 / `连接失败` 红）
   - 数据来自 `AIServiceStatus`（见 §7.2）

2. **Qwen 地区选择**（仅视觉 AI 需要，保留 CalorieCop）
   - Segmented：`国际` / `中国大陆`
   - 副文案：「仅影响 Qwen 识图服务器地址；DeepSeek 使用统一端点」

3. **DeepSeek API 区块**（必填）
   - 标题 + 副标题：「文字解析 · AI 顾问 · 报告辅助」
   - 获取指引：
     1. 访问 [platform.deepseek.com](https://platform.deepseek.com) 注册
     2. 控制台 → API Keys → 创建密钥
     3. 复制粘贴到下方
   - 「打开官网」按钮 → `https://platform.deepseek.com`
   - `SecureField` + 粘贴 / 显示 / 清除
   - **「测试连接」按钮**：发送最小化 ping 请求，反馈：成功 ✅ / Key 无效 / 网络超时 / 余额不足
   - 状态徽章：`用户已配置` / `开发者预设` / `未配置`

4. **Qwen API 区块**（可选，**保留 CalorieCop 原设计不变**）
   - 标题 + 副标题：「拍照识食（可选，不配仍可文字录入）」
   - 同上输入控件 + 测试连接

5. **隐私与安全说明卡**（保留并增强）
   - 图标 `lock.shield` + 三条要点：
     - Key 仅存本机 Keychain，不上传
     - AI 请求仅发送食物描述 / 脱敏报告文本，不含姓名身份证
     - 体检原始图片不上传云端（Vision 本地处理）

6. **高级选项**（阶段二出现，默认折叠）
   - Toggle：「启用 AI 辅助结构化体检报告（需联网，发送脱敏文本）」
   - 说明：开启后 Vision 结果可经 **DeepSeek** 整理，用户校对前可预览

7. **操作按钮**
   - 主按钮「保存设置」（莫兰迪主色 `#ADC8F5`）
   - 次按钮「清除所有密钥」（destructive，二次确认）

**交互状态矩阵：**

| 状态 | UI 表现 |
|------|---------|
| 未配置 DeepSeek | 饮食 Tab 顶部显示 `APIKeyPromptView`；首页橙色提示条 |
| 仅配 DeepSeek | 文字录入 + AI 顾问可用；拍照按钮显示「需 Qwen Key」轻提示 |

| 测试连接失败 | 输入框下方红色 inline 错误文案 + 重试 |
| 网络离线 | 功能内 Toast「网络不可用，已切换为手动模式」 |

### 3.6.8 DeepSeek API 技术规格（阶段一实现依据）

| 项目 | 值 |
|------|-----|
| 服务商 | DeepSeek（深度求索） |
| 申请地址 | [platform.deepseek.com](https://platform.deepseek.com) |
| API 端点 | `https://api.deepseek.com/v1/chat/completions` |
| 请求格式 | OpenAI Chat Completions 兼容 |
| 默认模型 | `deepseek-chat` |
| 认证方式 | `Authorization: Bearer <API_KEY>` |
| 流式输出 | AI 顾问支持 `stream: true`（SSE） |

**请求示例（食物录入）：**
```json
{
  "model": "deepseek-chat",
  "messages": [
    {"role": "system", "content": "<FoodParsingPrompt>"},
    {"role": "user", "content": "一碗米饭，两个鸡蛋"}
  ]

```

**与 Qwen 分工（不变）：**

| 能力 | 服务商 | 端点 |
|------|--------|------|
| 文字（食物/顾问/报告辅助） | DeepSeek | `api.deepseek.com` |
| 图像（拍照识食） | Qwen VL Plus | `dashscope.aliyuncs.com` 或国际站 |

**改造注意：**
- 文字请求/响应使用 OpenAI 标准 `ChatCompletion` 格式
- `AIAdvisorView` 流式解析使用 OpenAI SSE 格式（`data: {...}` 行）
- Qwen 相关代码（`parseFoodImage`、`qwenEndpoint`）**不做任何修改**


### 4.2 IntegratedTDEEEngine 计算逻辑

**输入：**
- HealthKit：`restingEnergy` + `activeEnergy`（当日或 7 日均值）
- UserGoal：当前体重、目标体重、目标日期
- HealthProfileEngine：体检约束（如尿酸高 → 提示低嘌呤，不直接改 TDEE，但影响 AI 建议）
- 可选：用户手动体脂率（HealthKit 体脂或手动录入）

**公式：**
```
TDEE = restingEnergy + activeEnergy × activityFactor（默认 1.0，可设置 0.9–1.2）
需减脂肪公斤 = currentWeight - targetWeight
每日赤字 = (7700 × 需减脂肪公斤) / 剩余天数（上限建议 750 kcal/天）
dailyBudget = TDEE - 每日赤字
```

**宏量分配：**
- 蛋白质 ≥ max(体重×1.6g, 健康档案建议下限)
- 脂肪 25–30% 总热量
- 剩余分配碳水

**输出联动：**
- 首页热量环：`remaining = dailyBudget - todayIntake + todayBurnAdjustment`
- 食物确认页：展示「录入后剩余 X 大卡」
- AI 顾问：注入 `dailyBudget`、`weeklyDeficitProgress`

### 4.3 HealthProfileEngine 评估逻辑

**单次报告导入后：**
1. Vision OCR 提取字段 → 用户强制校对 → 存入 SwiftData
2. 与同指标历史比对 → 生成趋势（↑ 恶化 / ↓ 改善 / → 持平）
3. 规则引擎匹配 → 生成 `RiskFlag`
4. 汇总为 `HealthSummary`（一页可读结论）

**初始规则库（阶段二，至少 10 条）：**

| 指标 | 触发条件 | 建议动作 |
|------|----------|----------|
| 肺结节 | ≥ 5mm 或增大 | 年度胸部 CT 复查，生成待办 |
| 低密度脂蛋白 | 连续两次超标 | 3–6 月复查血脂 |
| 空腹血糖 | ≥ 6.1 mmol/L | 建议内分泌科就诊 |
| 甲状腺功能 TSH | 超出参考范围 | 3 月复查甲功 |
| 血红蛋白 | 低于参考下限 | 建议血液科/补铁评估 |
| 尿酸 | > 420 μmol/L（女） | 饮食低嘌呤提示，融入 AI 顾问 |
| 肝功能 ALT/AST | 连续异常 | 1–3 月复查肝功能 |
| 子宫肌瘤 | 增大趋势 | 妇科 B 超复查计划 |
| 乳腺结节 | BI-RADS ≥ 4 | 立即就诊待办 |
| 骨密度 T 值 | ≤ -1.0 | 补钙与负重运动建议 |

> 规则以「提醒 + 复查建议」为主，**不做医学诊断**。所有风险卡片标注「仅供参考，请遵医嘱」。

### 4.4 WorkoutPlanEngine 输出格式

按当前周期阶段 + 减脂目标 + Watch 历史运动量，生成**本周结构化计划**：

```
示例（黄体期）：
  周一：瑜伽 30min（低强度）
  周三：快走 40min
  周五：普拉提 30min
  周六：休息或散步 20min
  目标：中等强度 120min / 抗阻 0次（黄体期降级）

示例（卵泡期）：
  周二：力量训练 45min
  周四：HIIT 20min
  周六：力量训练 45min
  周日：有氧 30min
  目标：中等强度 150min / 抗阻 2次
```

完成度从 HealthKit `workout` 自动匹配（类型 + 时长），在减脂页与首页卡片展示。

## 9. Cursor 开发约束

规则文件位于 `.cursor/rules/`（替代原 `.cursorrules` 草案）：

| 规则文件 | 作用域 |
|----------|--------|
| `bonne-sante-core.mdc` | 始终生效：架构原则、AI 分工、合规 |

| `architecture.mdc` | Services/ViewModels：分层与 Context |
| `ai-integration.mdc` | AI 相关文件：DeepSeek/Qwen 约束 |

**摘要：**
- 文字 AI 仅使用 DeepSeek；禁止使用 MiniMax API
- 禁止删除 CalorieCop 保留清单中的任何核心功能
- 所有健康与热量计算必须经 UnifiedHealthContext，禁止 View 层直接算 TDEE
- UI：系统字体 + SF Symbols；色板用 Theme.swift；卡片 .regularMaterial，圆角 16
- 间距 4 倍数网格；支持 Dark Mode
- 每个新 View 必须带 Preview + 模拟数据
- 体检 OCR 结果未经用户校对不得入库（isVerified = false 不同步到 Engine）
- 风险规则输出必须含「仅供参考，请遵医嘱」

## 11. 产品设计专业审查（Design Decision Chain）

> 审查方法：基于 UED 设计思维决策链，对 PRD v2.3 做系统性评估。

### 11.1 总体评分

| 维度 | 得分 | 简评 |
|------|------|------|
| 用户目标清晰度 | **88/100** | 核心诉求「健康数据×减脂融合」已上升为第一原则 |
| 信息架构 | **78/100** | 5 Tab 合理，但首页信息密度偏高，需分层 |
| 任务流程完整性 | **82/100** | 主路径清晰；Onboarding、空状态、错误流 v2.2 已补 |
| 视觉规范可执行性 | **85/100** | 莫兰迪色板明确；缺 Dark Mode 具体色值和语义色 Token 表 |
| AI 配置体验 | **90/100**（v2.2） | 能力矩阵 + 多入口 + 降级策略，达到工具型 App 标准 |
| 健康场景可信度 | **80/100** | 有免责声明；风险卡片需统一「非诊断」视觉语言 |
| **综合** | **84/100** | 可作为开发总纲；按下方行动项补齐后可达 90+ |

### 11.2 页面类型定性

| 页面 | 类型 | 设计策略 |
|------|------|----------|
| 首页仪表盘 | 数据型 + 工具型 | 一屏一焦点：热量环为主视觉，卡片组次级 |
| 健康档案 | 数据型 + 内容型 | 时间线 + 图表优先，摘要页用文字叙事降低焦虑 |
| 减脂 Tab | 工具型 | 高频操作（记录饮食）置于拇指热区 |
| 待办 | 列表型 | 日期排序，左滑操作，空状态引导导入 |
| AI 设置 | 工具型 | 步骤式，状态可见，避免技术术语堆砌 |
| AI 顾问 | 对话型 | 气泡简洁，输入框固定底部，流式输出 |

### 11.3 主要优点（保留）

1. **设计哲学与产品定位一致**：「温暖舒适」莫兰迪 + 女性健康，视觉策略选型正确。
2. **UnifiedHealthContext** 解决了多 Tab 数据割裂的典型 PRD 缺陷。
3. **三阶段 MVP** 让每阶段都有可交付价值，符合 Goal-Directed Design。
4. **CalorieCop 保留清单** 防止改造时功能回退，工程可执行性强。
5. **v2.3 AI 选型** DeepSeek（文字）+ Qwen（视觉）分工清晰，DeepSeek OpenAI 兼容格式降低改造风险。
